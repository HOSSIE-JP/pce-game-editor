# PCE / Super CD-ROM2 メモリバンク運用ルール

このメモは、PCE Game Editor の CD-ROM2 / VN runtime で使う HuC6280 MPR と llvm-mos section の割り当てを固定化するための引き継ぎルールです。

## 基本方針

- CD-ROM2 build では、背景 tiles/map、sprite pattern、ADPCM 本体のような大きい payload はまず `cd.dataFiles` に並べます。`cache load bg/sprite` は低位 System Card RAM の visual payload cache へ明示 asset を先読みし、表示 command は cache hit なら RAM→VRAM/BAT、miss なら従来どおり `cd_transfer_scratch` 経由で CD→VRAM/BAT へ転送します。ADPCM は ADPCM RAM または streaming 経路へ送ります。
- CPU が頻繁に読む小さい runtime data だけを RAM bank に置きます。表示 asset の実体は CD data file 優先です。
- `cd.dataFiles` は sector 64 以降に並ぶ前提で generated metadata に sector を埋め込みます。build は IPL program の後ろへ padding file を挟み、ISO 上の最初の data file が sector 64 から始まるようにします。padding サイズは固定ではなく、ELF build 後に `pce-mkcd -v` でプログラム像の実セクタ数を測定して `64 - program終端sector` で決めます（`finalizePceCdDataPadding()` / `parseMkcdFirstDataSector()`）。**RAM bank の配置やデータ量を変えて program 像のサイズが変わると、固定 padding では data 開始 sector がずれ、埋め込み済みの sector 参照（`pce_vn_font_data` / `pce_vn_scene_packs[]` / asset の `cd_data_ref`）が全部ずれて全画面が壊れます。** font tiles の CD streaming 化のように program サイズに影響する変更をしたら、必ず実 ISO で data 開始 sector が 64 のままか確認してください。
- VN runtime は `template/template_pce_vn_cd/src/pce_vn_runtime.c` が単一の実体です。project 側へ同期されるので、bank ルール変更は必ず template を直します。

## CD-ROM2 RAM Bank Map

| Bank | MPR | 主用途 | ルール |
|---:|---:|---|---|
| 128 | 2 | llvm-mos 既定の常駐 `.text` / `.rodata` | 起動・薄い制御・小さい rodata 用。大きい runtime や asset を押し込まない。CD BIOS や VN data access 後に常駐 metadata を読む場合は `map_resident_data()` で戻す |
| 129 | 3 | VN runtime の banked code | `PCE_RAM_BANK_AT(129, 3)` と `VN_BANKED_CODE` で command interpreter / VBlank wait / ADPCM 制御、scene pack cache load の中核を置く。asset data を置かない。meta on-demand では `execute_command()` を bank129 の out-of-line 関数に保ち、sprite descriptor の hot field は snapshot して使う |
| 112-119 | 6 | VN visual payload RAM cache（実験版） | `VN_ENABLE_VISUAL_PAYLOAD_CACHE 1` のとき、8KB x 8 page = 64KB の LRU payload cache として使う。BG tiles、BG map、sprite patterns を `{kind, assetIndex, part}` key で保持する。CD BIOS の直接転送先にはせず、CD→bank132 `cd_transfer_scratch`→低位RAM page の順でコピーする。evict / clear は metadata だけを無効化し、VRAM/BAT/SATB/表示状態は触らない |
| 120 | 6 | 予備 | visual cache payload には使わない。bank121 helper code と隣接させないため、標準 runtime では未使用 |
| 121 | 4 | VN visual cache helper code | `PCE_RAM_BANK_AT(121, 4)`。`visual_code.bin` を CD から CPU 0x8000 へ読み込み、`cache load bg/sprite` の低位RAM操作だけを行う。bank133 Path B overlay とは別物で、bank133 を上書きしない。payload page としては使わない |
| 122-127 | - | 予備 | 標準 runtime では未使用。visual cache payload の拡張に使う場合も bank121 code との分離を保つ |
| 130 | 4 | VN runtime の 2 本目の banked code | `PCE_RAM_BANK_AT(130, 4)` と `VN_BANKED_CODE2` で scene pack choice/switch reader、preload scan helper、raw BG map の行単位 CD 転送、Choice UI helper、PSG sequencer helper、CD-DA resume sector helper、Effect の palette 補間 helper、sprite animation cache helper、ADPCM buffer-fit helper を置く。asset data を置かない。message グリフコンポジタは Phase 2 で bank133 overlay へ退避済み |
| 131 | 5 | CD fallback の小さい CPU-readable data | 例外的な fallback 用。通常の画像・sprite・ADPCM payload には使わない。**コードは置けない**（System Card/CD-BIOS が slot5=0xA000 で実行されるため physical bank131 にコードを置くと暴走する） |
| 132 | 6 | VN generated data ＋ CD転送スクラッチ | `PCE_VN_DATA_SECTION`。sprite animation、variable 初期値、scene pack directory、font tiles の CD data ref (`pce_vn_font_data`) を置く。**font tiles 本体は bank132 に常駐させず CD data file (`assets/generated/vn/font.bin`) からストリーム**。scene script 本体も CD data file。font streaming で空いた領域に **`cd_transfer_scratch`(2KB) を `section(".ram_bank132")` で移設**し、逼迫する console_ram を空ける。大量 asset では BG / Sprite / ADPCM / PSG / CD-DA metadata を Catalog v2 (`assets/generated/meta/asset_meta.bin`) へ移し、bank132 常駐は `pce_editor_*_meta` の sector/count/slot だけにする。Catalog mode では短い PSG SFX も含めて pattern を CD data file (`assets/generated/psg/<id>.bin`) として扱い、再生時に bank134/bank135 へ展開する。末尾 0xd078-0xdfff はオーバーレイ blob の良性 LMA に使用する（下記）。**固定バッファ `cd_transfer_scratch`(2KB)/`message_glyph_cache_masks`(1632B) は `.ram_bank132_tail`(NOLOAD) としてこの良性 LMA window に重ね、読まれない overlay コピーの RAM を再利用する**ので、増える常駐メタデータは 0xc000-0xd078=4216B を丸ごと使える。これを超える規模の asset metadata は resident へ戻さず Catalog v2 で O(1) に保つ |
| 134 | 6 | 再生中の PSG 曲パターンバッファ前半。bank132 と slot6 を時分割 | IPL 非ロード。`play_psg_asset()` が `load_psg_pattern_cd()` で現在再生中の step pattern 前半を CD から bank134 RAM へ読み込み、`psg_apply_step_row()` は読む前に `pce_ram_bank134_map()` で MPR6=bank134 を張る。8byte/entry で 1024 event まで収容。System Card 非干渉 |
| 135 | 6 | 再生中の PSG 曲パターンバッファ後半。bank132/bank134 と slot6 を時分割 | IPL 非ロード。pattern が 1024 event を超える場合だけ `load_psg_pattern_cd()` が後半を bank135 RAM へ読み込み、`psg_apply_step_row()` が `pce_ram_bank135_map()` で後半を走査する。bank134+135 合計 16KB、最大 2048 event。曲数無制限（同時に RAM に載るのは 1 曲） |
| 133 | 4 | コードオーバーレイ（Path B）。bank130 と slot4 を時分割 | IPL は bank128-132 しか自動ロードしないため bank133 は CD data file (`assets/generated/vn/overlay.bin`) からブート時に `pce_cdb_cd_read` で CPU 0x8000 へストリーム。`VN_OVERLAY_CODE`(`section(".vn_overlay")`) タグの関数を置く。現行 runtime は message グリフコンポジタ（`draw_message_glyph_at` / `draw_message_next_glyph` / `draw_message_text` / `preload_message_glyph_masks` と helper 群）を置く。System Card 非干渉（bank131 と違い安全） |
Visual payload cache は低位 RAM 利用の実験版として標準有効です。`visual_code.bin` / `.vn_visual_code` は bank121 専用、payload は bank112-119 専用とし、旧案のように同じ bank を code と payload page で共有しません。`visual_code.bin` を抽出した後の main ELF では、`.vn_visual_code` section だけでなく対応する bank121 `PT_LOAD` program header も `neutralizeElfLoadSegments()` で無効化します。section を消すだけでは `pce-mkcd` の初期ロード表に bank121 helper が残り、System Card boot が `JUST A MOMENT...` で止まることがあります。切り戻しが必要な場合は `VN_ENABLE_VISUAL_PAYLOAD_CACHE` を戻す形で、command record や JSON schema は維持します。bank128/129/130 はまだ薄い余白しかないため、新しい helper を追加する前に既存 code path の削減、`VN_BANKED_CODE` / `VN_BANKED_CODE2` / `VN_OVERLAY_CODE` の付け替えを検討し、必ず `mos-pce-cd-clang -Wl,--print-memory-usage` か `llvm-size -A` で再確認してください。

## コードオーバーレイ（Path B / Phase B1）

3 つの常駐コードバンク（128/129/130 ＝計約24KB）を超えるエンジンコードを、未使用の物理 bank133 へ退避してブート時に CD からロードし、bank130 と MPR slot4 を時分割する機構。**追加・拡張の具体手順、検証手順、残課題は専用の引き継ぎガイド [pce-vn-overlay-pathb.md](pce-vn-overlay-pathb.md) を参照。**

- **配置タグ**: `pce_vn_runtime.c` で `VN_OVERLAY_CODE`（CD build では `__attribute__((noinline, section(".vn_overlay")))`、非 CD では空）を付けた関数がオーバーレイに入る。Phase B1 当時は CD RLE 展開 (`cd_rle_ref_to_vram` / `cd_rle_bg_map_ref_to_vram`) を退避し bank130 を約3.3KB 緩和した。Phase 2 後は message グリフコンポジタを overlay に置く。per-frame sprite 差分 refresh は VBlank 同期のため bank130 に戻した。
- **同一コンパイル**: オーバーレイは本体と同じ link に含めるため、llvm-mos の zero-page 仮想レジスタや常駐シンボルがそのまま解決される（B0 の別リンク方式が抱えた `R_MOS_ADDR8 out of range` 等の壁を回避）。
- **リンカ fragment**: `pce-vn-manager.js` が `src/generated/overlay_insert.ld`（`INSERT AFTER .ram_bank132`）を生成し、`-Wl,-T` で本体 link に差し込む。`.vn_overlay 0x8000 : AT(0x0184d078)` で **VMA=CPU 0x8000（slot4 実行アドレス）・LMA=bank132 末尾の良性アドレス**に置く（`VN_OVERLAY_LMA`）。さらに同 fragment が `.ram_bank132_tail 0xd078 (NOLOAD)` を生成し、**読まれない overlay 良性コピーの RAM に write-before-read の固定バッファを重ねて再利用**する。これで bank132 8KB のうち overlay tail に取られていた ~3.6KB 相当が常駐メタデータ用に空き、大規模 CD プロジェクトがスケールする。大きい PSG パターンは bank132 ではなく bank134 へ CD ストリームするので、resident metadata は overlay LMA より前に収める。
- **抽出と mkcd 回避**: link 後に `llvm-objcopy --only-section=.vn_overlay` で `overlay.bin` を抽出（予約サイズ 2 sector=4KB に pad）し、`--remove-section=.rela.vn_overlay` で**オーバーレイ内部 relocation テーブルだけを除去**する。pce-mkcd は ELF の relocation を再適用するため、オーバーレイの 0x8000 VMA を持つ reloc を残すと `File address 0x8001 out of range` で失敗する。lld が既に適用済みなので `overlay.bin` は完成形、mkcd へは除去後の elf を渡す。
- **CD sector 予約**: `overlay.bin` のサイズは link 前に固定予約（`VN_OVERLAY_RESERVED_SECTORS`）して `generateVnSources` が CD sector を確定し、link 後に実バイトで同サイズ上書きするので sector がずれない。実コードが予約を超えると build error。
- **ランタイム dispatch**: 常駐 bank128 の `call_overlay_*` / locked wrapper 経由で **bank133 を slot4 に map → 呼び出し → bank130 を復帰**する。VDC を触る overlay dispatcher は **IRQ lock → bank133 map → overlay → bank130 restore → IRQ unlock** の順で、bank switch 全体を IRQ mask 区間に入れる。**slot4 を共有する bank130/bank133 間の直接相互呼び出しは禁止**（オーバーレイ実行中は bank130 が見えない）。オーバーレイ関数は slot2/slot3（bank128/129）・inline ヘルパ・console_ram・CD BIOS のみ呼ぶこと。新しい関数を退避する前に `llvm-objdump -d --section=.vn_overlay` で slot4(0x8000-0x9fff) への非内部 JSR/JMP が無いことを確認する。
- **オーバーレイサイズ上限**: 良性 LMA が bank132 末尾（0xd078-0xdfff）にあるため `.vn_overlay` セクションの実バイトはこの window 以下。CD 上の予約は `VN_OVERLAY_RESERVED_BYTES`（4KB）のまま固定し、`finalizeOverlayBlob()` が予約サイズ超過と **bank132 末尾超過 (`VN_OVERLAY_LMA + realSize > VN_BANK132_LMA_END`)** の両方を検査し、超える場合は明示エラー。

## 実装ルール

- `ram_bank129` は起動時に `pce_ram_bank129_map()` で MPR3 へ常時 map します。`VN_BANKED_CODE` 関数はこの前提で直接呼びます。ADPCM の load/play/streaming 制御も bank128 を圧迫しないよう bank129 側へ置きます。bank129 は command loop / ADPCM / sprite refresh で膨らみやすいため、PSG の細かい helper や Effect の palette 補間、Choice/IF/Switch/Goto を含む command execution 本体など、bank130 から呼んでも問題ない処理を戻さないでください。meta on-demand では accessor 戻りポインタ由来の field load が大きくなりやすいので、`vn_get_*_asset()` はループ/関数入口で 1 回だけ呼び、必要 field を slot-local draw metadata、data ref、cell map pointer、snapshot state へ落としてから使います。
- `ram_bank130` も起動時に `pce_ram_bank130_map()` で MPR4 へ常時 map します。`VN_BANKED_CODE2` は bank129 に収まらない scene pack reader / preload helper / PSG helper / Effect palette helper / Choice UI helper / command execution helper 用で、MPR4 を asset fallback に切り替えない前提です。Choice/Variable/IF/Switch/Goto/Label/InputCheck を扱う `execute_control_command()`、`draw_choice_options()`、`jump_to_command()` は `VN_BANKED_CODE2` の noinline 関数として保ち、分岐処理や Choice UI が `run_commands_until_wait()` / bank128 `.text` へインライン展開されて各 bank を再び溢れさせないようにします。`draw_choice_options()` から message glyph を描く場合は resident dispatcher `call_overlay_draw_message_glyph_at()` 経由で bank133 overlay を呼びます。
- `ram_bank128` の常駐 data を読む runtime code は、CD BIOS helper や `map_vn_data()` 呼び出し後に `map_resident_data()` を挟んでから参照します。小規模 resident mode では `pce_editor_sprite_assets[]`、`pce_editor_sprite_draw_meta[]` のような metadata を bank128 resident data として扱います。**大量 asset では、ジェネレータが BG / Sprite / ADPCM / PSG / CD-DA metadata を CD data file `asset_meta.bin` へ逃がし、ランタイムの `vn_get_*_asset` accessor がオンデマンドでストリームします（[pce-asset-meta-cd-ondemand.md](pce-asset-meta-cd-ondemand.md)）。RAM 常駐は各種別の `pce_editor_*_meta` sector/count/slot と asset count だけにし、asset 数に対して O(1) に保ちます。** `pce_editor_*_asset_count` と `pce_editor_meta_region_t.count` は `unsigned int` です。runtime では scene command の signed index を `0..count-1` で検証してから使い、`(uint8_t)asset_index` 比較へ戻さないでください。
- runtime state の非ゼロ初期値は `.data` / `.zp.data` として bank128 の load 領域も消費します。`ui_text_color`、InputCheck の target など、`init_runtime_state()` や scene 入場で必ず初期化する値は BSS に置き、`= PCE_VN_NO_COMMAND` のような static initializer を戻さないでください。数 byte でも bank128 overflow の原因になります。
- 生成済み C metadata は scene 生成時に asset ID を index へ解決済みで、runtime では ID 文字列を使いません。`pce_editor_psg_asset_t` / `pce_editor_adpcm_asset_t` / `pce_editor_cdda_asset_t` に `id` field を戻すと bank128 の `.rodata` を直接圧迫するため、debug 用文字列は JSON 側に留めます。
- CD-ROM2 VN build では、`assets.c` に出す runtime asset metadata を scene command 参照済み asset に絞ります。未使用 BG/sprite/audio を `pce-assets.json` に残しても、scene から参照しない限り VRAM 排他予約・asset index・resident bank128 予算には入れません。asset index の整合性を保つため、`generateVnSources()` の index 作成と `generateAssetSources({ assetIds })` の filter は同じ asset ID set を使います。
- `ram_bank132` を読む前は `pce_vn_font_tiles_map()` または runtime の `map_vn_data()` を呼びます。MPR6 を切り替えるため、MPR6 上で実行される code を作らないでください。**MPR6(slot6) は bank132（VN data / catalog scratch）、bank134/bank135（再生中 PSG パターン）が時分割で共有**します。bank132 を読む側は使用直前に必ず `map_vn_data()`、PSG パターンを読む `psg_apply_step_row()` は前半で `pce_ram_bank134_map()`、後半で `pce_ram_bank135_map()` を張ります。BG/sprite/font/scene pack などの CD 転送待ち、RAM/CD→VRAM 転送、palette fade 中は `service_psg_during_blocking_work()` / `service_psg_during_blocking_frames()` が cooperative に `tick_psg()` を呼びます。CD read 1 chunk は `VN_PSG_CD_TRANSFER_COMPENSATION_FRAMES` で補償し、PSG 再生中の BG/sprite VRAM copy は `VN_VISUAL_VRAM_COPY_SLICE_BYTES` ごとに分割して各 slice 後に service します。PSG が鳴っていない場合は `VN_VISUAL_VRAM_COPY_FAST_SLICE_BYTES`（1 sector）までまとめて TIA 転送します。PSG pattern を読むと MPR6 が bank134/bank135 へ変わるため、通常の blocking helper は最後に `map_vn_data()` と `VN_MAP_BANK130_FOR_CODE()` を戻します。
- `cd_transfer_scratch`(1 sector=2KB) は `section(".ram_bank132")` に置き、CD→VRAM 転送ヘルパ（`cd_data_ref_to_vram` / `cd_bg_map_ref_to_vram` / `upload_font_tiles`）と visual cache の CD→低位RAM page 転送で使います。各ヘルパは asset/font の CD data ref を読む前に `map_vn_data()` で MPR6=bank132 を張り、CD read 完了後に CD-DA を再開してから RAM→VDC data port へ TIA 転送します。PSG 再生中だけ短い slice 単位で cooperative service を挟み、無音時は 1 sector までまとめて VRAM copy します。逆に **CD BIOS の書き込み先になる buffer や CD data ref を正しい MPR6 bank へ張らずに使うと、転送データや sector 参照が別バンクへ化けます**。`vn_active_scene_pack_data`(4KB) はコマンド解釈の各所から読むため console_ram に残します（bank132 へ移すなら全 reader で MPR6 を保証する必要があり要慎重）。
- グリフフォントは BG タイルと同じく `cd.dataFiles` の `assets/generated/vn/font.bin` に並べ、起動時に `upload_font_tiles()` が `cd_transfer_scratch` 経由で 1 回だけ VRAM (`PCE_VN_FONT_MASK_VRAM_WORD` から) へストリーム転送します。font.bin の中身は **12x12 1bpp マスク（12 word=24 byte/グリフ）**で、表示用の先焼きタイルではありません。CD runtime は message 開始時、ADPCM を鳴らす前にそのページの glyph mask だけを上限付き RAM cache へ先読みし、音声再生中の compositor は VRAM 読み戻しではなく cache を優先して 12px ピッチで合成します（cache 外 glyph だけ従来の VRAM 読みへ fallback）。合成時は tile と glyph の交差範囲だけを走査し、前 glyph が重ならない tile は即 return して、ADPCM 再生中の per-glyph CPU cost を抑えます。cache mask 本体は console RAM を圧迫しないよう `.ram_bank132` に置き、lookup 用 ID/count だけを `.bss` に残します。bank132 には sector/サイズだけを持つ `pce_vn_font_data` (`pce_vn_cd_data_ref_t`) を常駐させ、glyph 数に比例した数 KB を bank132 から外します。font.bin は `collectCdDataFiles()` で先頭 (CD sector 64) に置き、埋め込み sector と ISO 配置を一致させます。glyph index は 16bit エスケープ符号化（0..252 は 1 byte、253 以上は `0xfd`+16bit LE）で旧 254 種上限を超えられ、VRAM 側の上限だけが残ります。`generateVnSources()` の `computeFontBudget()` がマスク領域末尾 (SATB `0x7f00`)・VRAM tile soft ceiling をビルド時に検査し、超過は build error、接近は警告します（既定 `tileBase` で実用上限はおよそ 1000 種 = `VN_MAX_GLYPH_COUNT`）。マスク領域は固定 208 タイルのメッセージ帯＋blank tile の直後に置かれます。非 CD (`__PCE__`) build だけ従来通り `pce_vn_font_tiles[]`（中身はマスク）を埋め込みます。
- `spritetext` command を使う場合のみ、スプライト用フォント `assets/generated/vn/font_sprite.bin` を追加生成します（`spritetext` が実際に使う文字種だけを PCE スプライトパターン形式で encode）。font.bin と同じく `cd.dataFiles` に並べ、起動時に `upload_font_sprite_patterns()` が `cd_transfer_scratch` 経由で 1 回だけ **スプライト pattern VRAM (`PCE_VN_FONT_SPRITE_PATTERN_BASE`)** へストリームし、bank132 には `pce_vn_font_sprite_data` (sector/サイズ) だけ常駐させます。pattern base は BG グリフフォント直後（32-word pattern 単位）へ自動配置し、VN character sprite はそのさらに後ろの `PCE_VN_SPRITE_PATTERN_BASE` から連続配置します。spritetext font が SATB と重なる場合は `generateVnSources()` が warning を出します。色は予約 sprite palette bank `PCE_VN_FONT_SPRITE_PALETTE_BANK`（既定 15）の index15 を runtime が command ごとに書き換えて表現します。文字種ゼロ（spritetext 未使用）なら font_sprite.bin は生成せず、CD sector も予約しません。
- VN script は scene 単位の `assets/generated/vn/scenes/NNN_<sceneId>.bin` として `cd.dataFiles` に置きます。pack は pointer を持たない little-endian / offset ベース形式で、`pce_vn_scene_packs[]` だけを bank132 に常駐させます。
- runtime は scene 入場時に active scene cache (`4096` bytes) へ pack を読み込み、最初の `message` / `choice` / `wait` / `jump` までに必要な初期表示 asset だけを先読みします。別 scene 用の preload scan cache は持ちません。旧 `preload` command は廃止済みです。`cache` command の manual clear は BG/sprite/ADPCM/message glyph の runtime cache valid flag と visual payload cache metadata を落とすだけで、active scene pack cache や bank128/129/130/132/133 の code/data 配置とは別の制御です。`cache` command の manual load は scene pack を別途保持せず、command record に明示された単一 asset を対象にします。ADPCM は ADPCM RAM へ、BG/Sprite は bank112-119 の visual payload cache へ読み込みます。scene丸ごとの別pack先読みや二重scene cacheではありません。pack が 4096 bytes を超える場合は build error にして scene 分割を促します。
- scene pack の各 record は固定長です。command record は 19 byte、message record は 13 byte（`PCE_VN_SCENE_PACK_MESSAGE_SIZE`。末尾に文字色 `text_color` の 9-bit パレット word を u16 で持つ。未指定は `PCE_VN_MESSAGE_COLOR_NONE`=0xffff。mouth slot byte は下位 2bit が slot、上位 6bit が typewriter 前に即時表示する先頭 glyph entry 数）、choice record 6 byte、switch record 5 byte。この値を変える場合は `pce-vn-manager.js` の `encode*Record` / `VN_SCENE_PACK_*_SIZE`、`renderer.js` の `estimateScenePackBytes` 見積り定数、runtime の `scene_pack_read_*` オフセットと `pce_vn_*_t` struct、`vn.h` の `#define ...SIZE` を同時に合わせます。
- scene pack から読む command/message/choice/switch の要素は、CD / asset bank / VDC 転送で MPR が変わる可能性を考え、処理前に stack local にコピーしてから扱います。特に ADPCM は BIOS 呼び出し前に data size、address、divider、loop、stream、CD sector を runtime-owned snapshot へ直接コピーし、BIOS 呼び出し後に `pce_editor_adpcm_assets[]` のポインタを再読みに使わないでください。ADPCM meta / snapshot の multi-byte field や `pce_editor_cd_data_ref_t` は `memcpy` や構造体同士の連続 field copy に戻さないでください。llvm-mos が `tii $7c,$6a,#$4` や destination `$0089` のように高位アドレスを落とす転送へ畳むと、WRAM `$207c` / `$2089` ではなく zero page を読んだり書いたりして、ADPCM `data_size` や CD sector が 0 になります。生成ヘッダの `PCE_EDITOR_META_ADPCM_*` offset から scalar decode し、snapshot へは local scalar 経由で書きます。
- VN 変数 storage は 16bit 配列にしないでください。llvm-mos が `vn_variables[index]` を ZP indirect pointer に畳むと、PCE の I/O page 側へ書いて VDC data port 経由で VRAM を壊すことがあります。runtime は `vn_variable_lo[]` / `vn_variable_hi[]` の byte 配列を `VN_BANKED_CODE` の getter/setter で読み書きし、Choice の変数代入や IF/Switch の参照で indirect 16bit store を発生させないようにします。
- CD-ROM2 VN runtime では bank129 / bank130 / bank132 は runtime code / VN data 専用です。`pce_editor_map_asset_bank()` を使う banked asset fallback が必要な場合も、VN runtime では bank131 だけを例外的に使います。
- bank129 から bank130 の helper を呼ぶときは、caller の stack/local への writable pointer を渡さないでください。llvm-mos の banked 呼び出しで pointer が `$0002` などの I/O page に化けると、VDC data port へ書いて VRAM を壊します。CD-DA resume のような banked helper は戻り値か専用 BSS (`cdda_resume_start/end` など) で受け渡します。`VN_OVERLAY_CODE` (bank133) から呼ばれる可能性がある resume 系 helper は、overlay 実行中に bank130 が slot4 から外れるため、bank129 (`VN_BANKED_CODE`) か最小限の bank128 に置きます。
- CD-ROM2 の BG `map_vram.bin` は CD 上では `VN_MAP_WIDTH`(=32)タイル幅のソース行です（旧 64 幅から変更。`PCE_BG_MAP_WIDTH_TILES` と一致）。`mapBase` からファイル全体を一括転送すると、行末が次行左端へ回り込んで本来画像にない縦縞が出ます。runtime は raw `map_vram.bin` を `cd_transfer_scratch` へ1 sectorずつ読んで、各行の `width_tiles` 分だけを `mapBase + command.y * VN_MAP_WIDTH + command.x + row * VN_MAP_WIDTH` へコピーし、左右/上下余白は `clear_screen_map()` のblank tileを残します。
- sprite 描画に必要な cell size、sheet cell 数、pattern base、palette bank は generated `pce_editor_sprite_draw_meta[]` にも小さく出します。CD runtime は共有 `sprite_draw_meta` を持たず、`vn_get_sprite_asset()` から得た descriptor の cell metadata と、SLOT順に割り当てた pattern base / palette bank を slot-local draw metadata へスナップショットして SATB を組みます。animation metadata が sheet 範囲内なら `frame_count: 1` の default でも frame size として使います。旧 default のように frame size 未指定の asset は generator が sheet 全体の 1 frame へ補正します。
- sprite pattern は VRAM の sprite pattern 領域へ転送し、SATB は `VN_SATB_ADDR` (`0x7f00`) を使います。CD-ROM2 でも CD BIOS graphics driver は使わず、System Card の VBlank handler も使いません。通常 frame wait は `pce_cdb_wait_vblank()` の BIOS counter ではなく VDC status の `VDC_FLAG_VBLANK` を guard 付きで直接待ちます。`VN_VDC_CONTROL_BASE` には VDC status latch 用に `VDC_CONTROL_IRQ_VBLANK` を入れますが、HuC6280 側の `IRQ_VDC` は `pce_irq_disable(IRQ_VDC)` で mask します。`PCE_CDB_MASK_VBLANK_NO_BIOS` だけでは System Card の VBlank handler (`$E870`) が R5/R7/R8 を毎フレーム書くことがあるため、message 中の BG 水平ずれの原因になります。CD/ADPCM BIOS helper 後も `set_vdc_control()` で System Card の R5 shadow (`$F3/$F4`) を更新し、万一 BIOS helper が shadow を参照しても sprite bit が戻されないようにします。shadow SATB は直接 VRAM へ転送してから `VDC_REG_SATB_START` を維持します。未使用 SATB entry はゼロ化せず `VN_SPRITE_HIDDEN_Y` へ退避します（ゼロ entry も PCE では実 sprite として扱われ、BG の色0部分から pattern 0 が見えることがあります）。同一 slot へ別 sprite asset をロードする pattern upload では、sprite layer を無効化して画面外 SATB を反映し、VRAM 転送後に SATB と sprite layer を再有効化します。`sprite` command はこの一連の表示無効化 → VRAM 転送 → 表示有効化が完了してから次 command へ進みます。
- sprite animation の frame 変化だけでは full refresh を要求しません。full refresh 時に記録した slot ごとの SATB 開始位置と entry 数を使い、既存 layout のまま該当 frame を作り直して SATB の pattern / attr word だけを VRAM へ転送します。位置/表示/flip/animation 切替、spritetext 点滅、screen shake、pattern upload が絡む場合だけ full refresh とし、ADPCM 再生中の animation tick で `clear_sprites()`、palette upload、pattern CD load、64 entry 全転送を繰り返さないでください。ADPCM 再生中でも sprite/spritetext refresh は gate で止めず、口パクを動かすため slot に cache した animation metadata と pattern / attr word 差分更新で per-frame cost を抑えます。
- `background` command も同期 command です。BG 切替は Fade 前提で、`fadeOutFrames` / `fadeInFrames` は `10 / 20 / 30 / 40 / 50 / 60` のプリセットへ正規化します。未指定時は速度3の `30` です。保存済みの旧 `transition: "cut"` は読み込み時に `transition: "fade"` へ移行され、BG の VRAM/BAT 反映と fade 完了まで次 command へ進みません。fade は BG palette bank だけを段階変更し、message UI palette までは暗転させません。
- active message record は `start_message()` で `active_message_state` へコピーし、`tick_active_message()` と auto advance はこの cache を使います。ADPCM 再生中の typewriter tick で `scene_pack_read_message()` の range check / field decode を毎フレーム再実行しないでください。
- `delay_frame()` / `vn_wait_next_vblank()` の VBlank polling は inline asm の tight loop と短い guard で命令数/I/O read 数を抑えます。これは VBlank 数を数える timing 補正ではなく、表示期間中の SATB 更新を避けるための edge wait です。C の 16-bit guard loop や長い polling loop へ戻すと ADPCM 再生中に待ちループ自体が main loop を圧迫し、typewriter/counter が実時間から遅れます。
- VDC memory control は `VN_VDC_MEMORY_CONTROL` (`VDC_CYCLE_4_SLOTS | VDC_BG_SIZE_32_32`) を標準にします。画面は **256x224**（横ドットクロック 5.37MHz の 256 モード）、BAT は **32x32**（=32x30 タイルの表示領域に十分）です。BG size を設定し直す時に sprite cycle bit を落とすと sprite layer が見えなくなるため、`VDC_REG_MEMORY` へはこの定義を使ってください。BAT が 32 幅になったため `VN_MAP_WIDTH`(=32) と `pce-asset-manager.js` の `PCE_BG_MAP_WIDTH_TILES`(=32) は一致させ、BG 画像は **256px(32 タイル)以下**にしてください（`encodePceBackground` が超過時にエラーにします）。
- `pce_editor_vram_copy()` は resident/noinline の共通 VDC blit helper とし、内部で `vn_vdc_irq_lock()` / `vn_vdc_irq_unlock()` を行います。message window clear、`write_map_words()` の BAT 行更新、raw BG/map/font/sprite pattern 転送はすべてこの helper を通すこと。ADPCM/CD external IRQ が MAWR 設定から VRAM data 転送の途中に入ると、数フレームだけ BG/メッセージが壊れて戻るランダムな表示崩れになります。message 開始時の strip clear、速度0の初回全文表示、choice 再描画の一括 glyph draw は、表示中に参照中の 208 tile を連続更新しないよう、message 窓 BAT を `PCE_VN_BLANK_TILE` へ一時退避してから strip tile を更新し、最後に strip BAT へ戻します。Message 表示途中のボタンスキップはこの退避を使わず、現在の glyph cursor から残りを追加描画して既存テキストを消さないでください。Message 用には全画面 display blank や `pending_display_enable` を使わないでください。VN runtime では `pce_vdc_set_copy_word()` を使わず、R5 low byte を保ったまま high byte だけ `VN_VDC_CONTROL_BASE >> 8` へ戻す `vn_vdc_set_copy_word()` を使います。SDK helper は R5 high byte を 0 にして DRAM refresh / VBlank status latch を落とすため、表示崩れや VBlank wait の timeout を再発させます。
- ADPCM / CD-DA の CD BIOS helper から戻った後は、R5 だけでなく `pce_vdc_set_resolution(256, 224, VCE_COLORBURST_ON)`、`VDC_REG_MEMORY`、SATB start、scroll を runtime 標準値へ戻します。System Card BIOS は ADPCM 処理中に水平/垂直 timing register を触ることがあり、display enable だけの復元では画面が上下に二重表示されることがあります。この復元シーケンスも VDC register-select latch を触るため、`restore_video_after_cdb_call()` 内で VBlank へ寄せてから IRQ mask したまま実行します。単独で呼ばれる display/sprite layer control は VBlank wait 経由で R5 を更新し、screen-shake scroll 更新でも latch 競合が起きるため `set_vdc_control()` と `apply_screen_offset()` 自体も IRQ guard 付きにします。
- CD-DA 再生中の scene pack / BG / sprite / font / ADPCM などの有限 CD data read は、`prepare_cd_data_access()` で CD-DA を一時 pause します。CD-DA と CD data read は同じドライブを使うため同時実行できませんが、BG/sprite の raw visual payload は各 sector read 後に `finish_cd_data_read_before_vram_copy()` で `resume_cdda_after_cd_data_access()` と `map_vn_data()` を済ませてから `cd_transfer_scratch` → VRAM/BAT copy へ入ります。これにより、タイル/マップの VRAM 書き換え中まで CD-DA pause を引きずりません。scene pack など、まとまったロード区間では引き続き `begin_cdda_deferred_resume()` / `end_cdda_deferred_resume()` で再開を遅延します。CD-DA をより途切れさせたくない scene では、BG/Sprite の表示 command 自体を CD-DA 開始前へ寄せてください。CD から直接読み続ける ADPCM stream はドライブ競合になるため、`cancel_cdda_after_cd_data_conflict()` で CD-DA 継続状態を破棄します。
- System Card external IRQ は CD data read / CD-DA pause/play / ADPCM load/stop/reset など BIOS helper の直前だけ有効化し、helper 後は `sync_cd_external_irq_after_bios_call()` で真の ADPCM stream 中以外は無効化します。さらに System Card の VBlank handler は runtime では使いません。Geargrafx では `PCE_CDB_MASK_VBLANK_NO_BIOS` 有効時でも System Card の VBlank handler が R5/R7/R8 を書くことを確認済みで、通常 message frame にこの IRQ を残すと BG が一瞬水平にずれる原因になります。VDC R5 の VBlank bit は polling 用に有効化し、HuC6280 `IRQ_VDC` は mask する、という分離を守ってください。

## PSG 補償 tick 調整 TIPS

BG 切替や sprite/font/scene pack の同期 CD ロード中は main loop に戻らないため、PSG は `service_psg_during_blocking_work()` / `service_psg_during_blocking_frames()` で cooperative に進めます。調整対象は `template/template_pce_vn_cd/src/pce_vn_runtime.c` の次の 2 定数です。

- `VN_PSG_CD_TRANSFER_COMPENSATION_FRAMES`: CD read 1 chunk (`cd_transfer_wait()` / visual cache CD read wait) ごとの補償 tick。BG 切替中の PSG 全体が遅く聞こえる場合はまずこちらを上げます。既定は `20u`。
- `VN_VISUAL_VRAM_COPY_SLICE_BYTES`: PSG 再生中の BG/Sprite RAM/CD buffer→VRAM copy を分割する byte 数。標準は Geargrafx 確認後 `64u` です。PSG が鳴っていない場合は `VN_VISUAL_VRAM_COPY_FAST_SLICE_BYTES`（既定 1 sector）までまとめて転送します。CD-DA/PSG の途切れが残る場合はこの値をさらに小さくできますが、総転送時間は伸びます。

調整の目安: CD read 待ちで PSG 全体が遅く聞こえるなら `CD_TRANSFER` を `+2..+4` ずつ上げます。VRAM 転送中の瞬間的な音切れや CD-DA 停止感が残る場合は `VN_VISUAL_VRAM_COPY_SLICE_BYTES` を小さくします。通常 frame loop の `tick_psg()` は通常再生テンポそのものなので、BG ロード中だけの補正では触らないでください。大きい PSG pattern は tick 中に MPR6 を bank134/bank135 へ切り替えるため、補償 helper の最後で `map_vn_data()` と `VN_MAP_BANK130_FOR_CODE()` を戻すルールは維持してください。
- メッセージフォントは **12x12 グリフ**で、1 行 17 文字 x 4 行（横 12px ピッチ・縦 16px 行ピッチ、メッセージ窓 208x64px を BAT (3,20) に配置）。12px ピッチは 8x8 タイル境界に乗らないため、`font.bin` は先焼きタイルではなく **12x12 1bpp マスク（12 word=24 byte/グリフ）**を持ち、起動時に `upload_font_tiles()` が `PCE_VN_FONT_MASK_VRAM_WORD`（=`(PCE_VN_FONT_TILE_BASE+208+1)*16`）以降の VRAM へストリームします。runtime のグリフコンポジタ（`draw_message_glyph_at` 他）は bank133 overlay (`VN_OVERLAY_CODE`) に置き、resident dispatcher から IRQ を mask した状態で bank133 を slot4 に map して呼び、bank130 へ復帰してから IRQ を戻します。overlay entry は先頭で `map_vn_data()` を呼び、message 開始時に先読みした `.ram_bank132` cache の mask を優先し、cache 外 glyph だけ `pce_vdc_copy_from_vram` で VRAM から読み戻して、メッセージ帯（`PCE_VN_FONT_TILE_BASE` から 208 タイル）の該当 8x8 タイルを read-modify-write で更新します。`vn_glyph_decode()` / `vn_glyph_stride()` は choice と overlay の共有 helper なので `VN_RESIDENT_CODE` に置きます。**全フォント分の常駐ピクセルバッファは持たず、コンポジタを bank128 に置かない**こと。点灯ピクセル=color15（4 プレーン全点灯）なので、タイル先頭 8 word の下位バイトが現在のマスクとして読み戻せます。BG/UI の blank tile は専用の `PCE_VN_BLANK_TILE`（メッセージ帯直後の 1 タイル、起動時にゼロ化）を使い、メッセージ帯タイルとは分離します。

## 変更時の確認

- runtime / bank 変更後は `npm test -- --test-name-pattern "PCE VN"` を最低限実行します。
- CD image を作れる環境では VN template を build し、Geargrafx MCP で VDC control、SATB、sprite pattern VRAM、sprite palette を確認します。
- Geargrafx MCP が使えない場合は、その理由を書いたうえで EmulatorJS / Test Play screenshot を補助確認にします。
